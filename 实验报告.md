## 一、研究背景与意义

### 1.1 研究背景

谷物是人类生存的基础粮食，包括大米、小麦、玉米、大豆等。谷物品质检测是粮食加工、储存、贸易中的重要环节。传统的人工目测方法存在效率低、主观性强、人力成本高等问题，已无法满足现代粮食产业对品质管控的需求。

随着计算机视觉和深度学习技术的飞速发展，利用目标检测算法进行谷物品质自动化检测已成为研究热点。通过工业相机采集谷物图像，结合深度学习模型实现端到端的品质分级检测，可大幅提升检测效率和客观性。

### 1.2 研究意义

- **学术意义**：探索RTMDet模型在细粒度农产品品质检测任务上的应用效果，验证CSPNeXt骨干网络在小目标检测上的性能
- **经济意义**：替代人工检测，降低粮食加工企业的人力成本
- **社会意义**：保障粮食安全，防止不合格谷物进入消费者市场

## 二、研究现状

### 2.1 传统谷物检测方法

| 方法 | 原理 | 优缺点 |
|------|------|--------|
| 人工目测 | 质检员肉眼观察 | 简单但效率低，主观性强 |
| 筛分法 | 不同孔径筛网筛选 | 只能检测粒度，无法检测品质 |
| 光电色选 | 颜色传感器分选 | 仅能区分颜色差异 |
| 机器视觉(SVM) | 手工特征+SVM分类 | 特征提取依赖经验，泛化性差 |

### 2.2 深度学习方法

近年来，基于CNN的目标检测模型在农业领域取得显著进展：

- **YOLOv5/YOLOv8**：单阶段检测器，速度快，适合实时检测场景
- **Faster R-CNN**：两阶段检测器，精度高但速度较慢
- **RTMDet (本项目选用)**：在精度和速度间取得最佳平衡

### 2.3 本项目创新点

1. 将RTMDet-tiny首次应用于谷物品质多分类目标检测任务
2. 设计四分类品质分级体系（完整粒/破损粒/霉变粒/异色粒），覆盖主要品质缺陷类型
3. 利用CSPNeXt骨干网络的通道注意力机制增强细粒度特征提取能力

## 三、模型设计

### 3.1 整体架构

采用 RTMDet-tiny 作为基线模型，基于 MMDetection 3.x 框架实现。

### 3.2 骨干网络 (Backbone)：CSPNeXt-tiny

CSPNeXt 是 ConvNeXt 的改进版本，通过跨阶段部分连接降低计算量同时保持特征提取能力。

### 3.3 特征金字塔 (Neck)：CSPNeXtPAFPN

采用路径聚合特征金字塔网络，通过自顶向下和自底向上的双向特征融合增强多尺度检测能力。

### 3.4 检测头 (Head)：RTMDetSepBNHead

解耦检测头，分类和回归分支独立，使用 Quality Focal Loss 和 GIoU Loss。

### 3.5 训练策略

- 优化器：AdamW, lr=0.004, weight_decay=0.05
- 学习率调度：Linear warmup (1000 iters) + Cosine Annealing
- 数据增强：Mosaic + MixUp + RandomResize + HSV增强 + RandomFlip
- 两阶段训练：前280 epoch强增强 → 后20 epoch弱增强微调
- EMA：指数移动平均，momentum=0.0002

## 六、遇到的 Bug 以及调试方案

### Bug 1: PyTorch CPU版本无法使用GPU训练

**现象**：
```
OSError: [WinError 1114] 动态链接库(DLL)初始化例程失败。
Error loading "c10.dll" or one of its dependencies.
```
执行 `torch.cuda.is_available()` 返回 `False`，模型只能在 CPU 上运行，训练速度极慢（约3.5秒/迭代，300 epoch 需11小时）。

**原因分析**：
conda 环境中安装的 PyTorch 为 `2.4.1+cpu` 版本（通过清华 pip 镜像安装时，Windows 平台默认拉取 CPU 版本），不含 CUDA 运行时库。此外，系统环境变量 `CUDA_HOME` 未设置，conda 默认 channel 被墙（`repo.anaconda.com` 返回 HTTP 403），无法通过 conda 直接安装 CUDA 版 PyTorch。

**解决方案**：
1. 排查 GPU 驱动：`nvidia-smi` 确认 RTX 3060 6GB 正常，驱动版本 610.47，CUDA UMD 13.3
2. 从 PyTorch 官方源下载 CUDA 预编译 wheel：
   ```bash
   pip install torch==2.1.0+cu118 torchvision==0.16.0+cu118 -f https://download.pytorch.org/whl/cu118/torch_stable.html
   ```
   注意：`torch==2.1.0` 是适配 Python 3.8 的最高 CUDA 版本，`torch==2.4.1+cpu` 不支持 CUDA。
3. 验证安装：`torch.cuda.is_available()` 返回 `True`，识别 `NVIDIA GeForce RTX 3060 Laptop GPU`

### Bug 2: MMCV CUDA扩展DLL加载失败

**现象**：
```
ImportError: DLL load failed while importing _ext: 找不到指定的程序。
```
即使 PyTorch 已正确安装 CUDA 版本，mmdet 初始化时仍报 `mmcv.ops.roi_align` 导入失败。

**原因分析**：
通过清华 pip 镜像（`pypi.tuna.tsinghua.edu.cn`）安装的 `mmcv==2.1.0` 是源码编译的 CPU 版本，其 C 扩展（`_ext.pyd`）未链接 CUDA 运行时库。mmdet 的 `structures/mask/structures.py` 需要调用 `mmcv.ops.roi_align`，因此触发 DLL 加载错误。

**解决方案**：
1. 从 OpenMMLab 官方下载针对特定 CUDA/PyTorch/Python 版本的预编译 wheel：
   ```
   https://download.openmmlab.com/mmcv/dist/cu118/torch2.1.0/mmcv-2.1.0-cp38-cp38-win_amd64.whl
   ```
   注意版本对应关系：`cu118` → CUDA 11.8，`torch2.1.0` → PyTorch 2.1.0，`cp38` → Python 3.8，`win_amd64` → Windows 64位。
2. 本地安装并强制覆盖：
   ```bash
   pip install mmcv-2.1.0-cp38-cp38-win_amd64.whl --force-reinstall
   ```
3. 同时修复 torchvision 版本冲突：
   ```bash
   pip install torchvision==0.16.0+cu118 -f https://download.pytorch.org/whl/cu118/torch_stable.html
   ```

### Bug 3: Windows GBK编码导致JSON加载失败

**现象**：
```
UnicodeDecodeError: 'gbk' codec can't decode byte 0x80 in position 104: illegal multibyte sequence
```
使用 `mmengine.Config.fromfile()` 或 `json.load(open(path))` 读取标注文件时崩溃。

**原因分析**：
在中文 Windows 系统中，Python 的默认文件编码为 GBK（`locale.getpreferredencoding()` 返回 `'cp936'`）。标注 JSON 文件 `instances_train.json` 中包含中文字段（如 `"description": "谷物颗粒品质分级检测数据集 - 合成图像"`），且文件未包含 BOM 头。当 `open()` 以默认 GBK 编码读取 UTF-8 文件时，遇到字节 `0x80` 等 UTF-8 多字节序列导致解码失败。

**解决方案**：
在 `train.py` 最顶部设置环境变量，强制 Python 使用 UTF-8 模式：
```python
os.environ['PYTHONUTF8'] = '1'
os.environ['PYTHONIOENCODING'] = 'utf-8'
```
`PYTHONUTF8=1` 是 Python 3.7+ 引入的 UTF-8 模式开关，会将默认编码从 GBK 切换为 UTF-8，无需修改每一个 `open()` 调用。

### Bug 4: mmengine环境检测MSVC编码崩溃

**现象**：
```
UnicodeDecodeError: 'utf-8' codec can't decode byte 0xd3 in position 0: invalid continuation byte
```
训练启动时，`mmengine.runner.Runner.__init__` → `_log_env()` → `collect_env()` 崩溃。

**原因分析**：
`mmengine/utils/dl_utils/collect_env.py:141` 中调用 `subprocess.check_output` 执行 MSVC 编译器获取版本信息，返回的字节流是 GBK 编码。源码尝试用 `locale.getpreferredencoding()` 解码，但设置了 `PYTHONUTF8=1` 后，`locale.getpreferredencoding()` 返回 `'utf-8'`，用 UTF-8 解码 GBK 字节流导致失败。

**解决方案**：
修改 `mmengine/utils/dl_utils/collect_env.py`，为 `cc.decode()` 添加多编码回退机制：
```python
encoding = os.device_encoding(sys.stdout.fileno()) or locale.getpreferredencoding()
try:
    env_info['MSVC'] = cc.decode(encoding).partition('\n')[0].strip()
except (UnicodeDecodeError, LookupError):
    for enc in ['gbk', 'utf-8', 'latin-1']:
        try:
            env_info['MSVC'] = cc.decode(enc).partition('\n')[0].strip()
            break
        except (UnicodeDecodeError, LookupError):
            continue
    else:
        env_info['MSVC'] = 'n/a (decode error)'
```
先尝试 GBK（Windows MSVC 输出的真实编码），再尝试 UTF-8 和 latin-1。

### Bug 5: TensorBoard依赖TensorFlow导致依赖崩溃

**现象**：
```
AttributeError: module 'numpy' has no attribute 'object'.
`np.object` was a deprecated alias for the builtin `object`.
```
导入 mmdet 时，TensorBoard 的 lazy import 触发了 TensorFlow 加载，而 TensorFlow 2.4.0 与 numpy 1.24.4 不兼容（使用了已废弃的 `np.object`）。

**原因分析**：
mmdet 配置中启用了 `TensorboardVisBackend`，会触发 `import tensorboard`。kereas 环境中残留的 `tensorflow 2.4.0` 依赖旧版 numpy API（`np.object` 在 numpy 1.24 中被移除），形成链式导入崩溃。

**解决方案**：
1. 卸载不兼容的旧版包：
   ```bash
   pip uninstall tensorflow tensorflow-estimator tensorboard -y
   ```
2. 在配置文件中移除 TensorBoard 后端，仅保留 `LocalVisBackend`：
   ```python
   vis_backends = [dict(type='LocalVisBackend')]
   ```
   训练日志通过 mmengine 的 LoggerHook 输出，不影响训练可视化。

### Bug 6: 子进程找不到项目内mmdet模块

**现象**：
```
ModuleNotFoundError: No module named 'mmdet'
```
使用 `subprocess.run()` 启动子进程执行 `tools/train.py` 时，子进程无法导入项目内部的 `mmdet/` 包。

**原因分析**：
原版 `train.py` 通过设置 `PYTHONPATH` 指向外部 mmdetection 仓库来导入 mmdet。改为自包含项目后，`mmdet/` 目录在项目根目录下，`sys.path` 默认不包含项目目录，且子进程不会继承父进程的 `sys.path` 修改。

**解决方案**：
放弃 `subprocess.run()` 子进程方案，改为直接在进程内导入并调用 `tools/train.py` 的 `main()` 函数：
```python
if PROJECT_DIR not in sys.path:
    sys.path.insert(0, PROJECT_DIR)
os.chdir(PROJECT_DIR)
from tools.train import main as train_main
train_main()
```
这样编码设置和路径配置在同一个进程中生效，避免了跨进程传递问题。
